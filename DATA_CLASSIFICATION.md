# Data Classification Policy

## Overview

This document describes the types of data processed by the Dyslexia Support Skill and Dyscalculia Support Skill, their classification levels, and handling requirements.

## Data Types and Classifications

### User Input Text (Classification: GENERAL / PUBLIC)

All text submitted to skill commands (e.g., text for simplification, math expressions for visualization) is treated as transient input. The skills do not persist, store, or log user input beyond the scope of a single request-response cycle.

**Handling requirements:** No special storage or encryption requirements. Input is passed to the configured LLM endpoint and discarded after the response is returned.

### LLM Responses (Classification: GENERAL / PUBLIC)

Output generated by the LLM (Ollama, OpenAI-compatible endpoints) is returned directly to the caller. No responses are cached, stored, or used for training.

**Handling requirements:** Same as user input — transient, not persisted.

### Configuration Data (Classification: INTERNAL)

Manifest files (`source/manifest.json`), locale strings (`source/i18n/*.json`), and skill/command markdown files contain the prompt engineering and parameter schemas that define each command's behavior.

**Handling requirements:** Version-controlled in the repository. No secrets or credentials should be stored in these files. Review changes via pull request.

### API Credentials (Classification: CONFIDENTIAL)

Ollama endpoint URLs, API keys (for OpenAI-compatible endpoints), and any authentication tokens are confidential.

**Handling requirements:**
- Never hardcode in source files — use environment variables (`OLLAMA_URL`, `OLLAMA_MODEL`, `OPENAI_API_KEY`)
- Never commit to version control
- Store in `.env` files (excluded via `.gitignore`) or secret managers
- Sample code uses placeholder values (`not-required-for-ollama`) to indicate where real credentials belong

### Build Artifacts (Classification: PUBLIC)

Generated output in `dist/` directories (Claude plugins, OpenAI function schemas, n8n nodes, YAML prompts, MCP servers, CLI tools) is derived entirely from the public configuration data and contains no secrets.

**Handling requirements:** Safe to distribute. Included in npm packages and plugin archives.

## Data Flow Summary

```
User Input (GENERAL) → LLM API (via CONFIDENTIAL credentials) → LLM Response (GENERAL)
                                                                        ↓
                                                                  Returned to caller
                                                                  (not persisted)
```

## PII and Sensitive Data

These skills are designed for educational accessibility support. However, user-submitted text could potentially contain personally identifiable information (PII) such as student names or personal writing samples.

**Mitigations:**
- No data is logged, stored, or transmitted beyond the configured LLM endpoint
- When using Ollama locally, all processing stays on-device
- When using cloud LLM endpoints (OpenAI, etc.), users should review that provider's data retention and privacy policies
- No telemetry, analytics, or usage tracking is implemented

## Compliance Notes

This project does not collect, store, or process regulated data (HIPAA, FERPA, COPPA, GDPR). Users who process regulated data through these skills are responsible for ensuring their LLM endpoint configuration meets applicable compliance requirements.
